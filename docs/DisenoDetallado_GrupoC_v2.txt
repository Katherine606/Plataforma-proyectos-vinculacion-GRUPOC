1. INTRODUCCION
================================================================================

El presente documento describe el disenio detallado del sistema "Plataforma de
Gestion de Proyectos de Vinculacion", una aplicacion web construida bajo la
arquitectura MVC (Model-View-Controller) utilizando Angular 21 en el frontend,
PHP 8.0 en el backend y MySQL como sistema de gestion de bases de datos.

El objetivo es centralizar, automatizar y auditar los procesos de gestion de
proyectos de vinculacion comunitaria de la universidad, reemplazando procesos
manuales y dispersos.


2. OBJETIVOS
================================================================================

2.1 Objetivo General

  Diseniar e implementar una plataforma web que permita gestionar de forma
  integral los proyectos de vinculacion comunitaria de la universidad,
  incluyendo autenticacion, CRUD de proyectos, inscripcion de estudiantes,
  registro y aprobacion de horas, y dashboards de seguimiento.

2.2 Objetivos Especificos

  OE1: Implementar un sistema de autenticacion con JWT y roles (Coordinador,
       Tutor, Estudiante) que controle el acceso a cada modulo.

  OE2: Desarrollar el modulo de CRUD de proyectos con validaciones en
       frontend y backend, incluyendo gestion de cupos y fechas.

  OE3: Implementar el motor de inscripcion con validacion de cupos en
       tiempo real y regla de un solo proyecto por estudiante.

  OE4: Desarrollar el flujo de registro y aprobacion de horas de vinculacion
       vinculadas a actividades creadas por el Tutor.

  OE5: Implementar dashboards personalizados por rol con graficos de barras
       y barras de progreso.

  OE6: Garantizar la integridad de datos mediante transacciones MySQL,
       sentencias preparadas y normalizacion en 3FN.

5. MODULOS DEL SISTEMA 
================================================================================

5.1 Modulo 1: Autenticacion y Seguridad

  Funcionalidades:
    - Login con correo y contrasena.
    - Generacion de token JWT (HMAC-SHA256, TTL 2 horas).
    - Guard de ruta (AuthGuard) que valida el token.
    - Guard de rol (RoleGuard) que controla acceso por rol.
    - Interceptor HTTP que agrega el header Authorization.
    - Sesion persistente via localStorage (token) y sessionStorage (usuario).
    - Logout que limpia ambos almacenamientos.

  Flujo:
    1. Usuario ingresa credenciales.
    2. Frontend envia POST a ?recurso=auth&accion=login.
    3. Backend verifica correo + bcrypt + estado activo.
    4. Se genera JWT con payload: {sub, nombre, apellido, correo, rol, rol_id, exp}.
    5. Frontend guarda token y usuario, redirige segun rol.

5.2 Modulo 2: Gestion de Proyectos (CRUD)

  Funcionalidades:
    - Coordinador: crear, editar, eliminar proyectos.
    - Todos los roles: listar proyectos con busqueda y paginacion.
    - Expandir detalle de proyecto (estudiantes inscritos, fechas, etc.).
    - Coordinador: remover estudiante del proyecto.
    - Estudiante: inscribirse a un proyecto.
    - Tutor: solo ve sus propios proyectos (filtro automatico).

  Endpoints:
    GET    ?recurso=proyectos              (listar, filtrado por rol)
    GET    ?recurso=proyectos&id=N         (detalle)
    GET    ?recurso=proyectos&accion=tutor (proyectos del tutor)
    POST   ?recurso=proyectos              (crear, Coordinador)
    PUT    ?recurso=proyectos&id=N         (editar, Coordinador)
    DELETE ?recurso=proyectos&id=N         (eliminar, Coordinador)
    PUT    ?recurso=proyectos&id=N&accion=sacar-estudiante

  Validaciones del formulario:
    - Nombre: obligatorio, solo letras/espacios, minimo 2 palabras.
    - Descripcion: obligatoria.
    - Facultad, carrera, tutor: obligatorios.
    - Cupos: entre 1 y 60.

5.3 Modulo 3: Inscripcion y Solicitudes

  Funcionalidades:
    - Estudiante envia solicitud de inscripcion.
    - Tutor/Coordinador acepta o deniega solicitudes.
    - Vista "Mis solicitudes" para el estudiante.
    - Tutor solo ve solicitudes de sus proyectos.
    - Validacion: un proyecto por estudiante.
    - Validacion: sin duplicados.
    - Transaccion MySQL al aceptar (incrementar cupos).

  Endpoints:
    GET    ?recurso=solicitudes                    (listar, filtrado por rol)
    POST   ?recurso=solicitudes                    (crear, Estudiante)
    PUT    ?recurso=solicitudes&id=N&accion=aceptar
    PUT    ?recurso=solicitudes&id=N&accion=denegar

5.4 Modulo 4: Actividades

  Funcionalidades:
    - Tutor crea actividades con titulo, fecha y descripcion.
    - Tutor ve lista de actividades con estadisticas.
    - Tutor expande actividad para ver estudiantes que registraron horas.
    - Tutor elimina actividades propias.
    - Estudiante ve actividades de sus proyectos aceptados.

  Endpoints:
    GET    ?recurso=actividades                    (listar)
    GET    ?recurso=actividades&accion=detalle     (detalle con stats, Tutor)
    GET    ?recurso=actividades&accion=estudiantes&id=N
    POST   ?recurso=actividades                    (crear, Tutor)
    DELETE ?recurso=actividades&id=N               (eliminar, Tutor)

5.5 Modulo 5: Registro y Aprobacion de Horas

  Funcionalidades:
    - Estudiante registra horas (1-8 por registro) vinculadas a una actividad.
    - Tutor aprueba o rechaza registros de horas.
    - Estudiante ve resumen y historial de horas.
    - Tutor ve horas pendientes de sus proyectos.

  Endpoints:
    GET    ?recurso=horas                    (listar, filtrado por rol)
    GET    ?recurso=horas&accion=resumen     (resumen del estudiante)
    POST   ?recurso=horas                    (registrar, Estudiante)
    PUT    ?recurso=horas&id=N&accion=aprobar   (Tutor)
    PUT    ?recurso=horas&id=N&accion=rechazar  (Tutor)

5.6 Modulo 6: Dashboard

  Funcionalidades:
    - Panel de inicio personalizado por rol.
    - Coordinador: estadisticas globales + grafico de solicitudes.
    - Tutor: estadisticas de sus proyectos + grafico de horas.
    - Estudiante: estadisticas personales + grafico de progreso.
    - Saludo personalizado segun hora del dia.
    - Tarjetas clickeables con enlaces a modulos.

5.7 Modulo 7: Gestion de Usuarios

  Funcionalidades:
    - Coordinador: CRUD completo de usuarios.
    - Tabla con nombre, correo, rol y estado.
    - Modal para crear/editar usuarios.
    - Toggle activar/desactivar usuario.
    - Eliminacion permanente.

  Endpoints:
    GET    ?recurso=usuarios                   (listar)
    GET    ?recurso=usuarios&accion=roles      (catalogo de roles)
    POST   ?recurso=usuarios                   (crear)
    PUT    ?recurso=usuarios&id=N              (editar)
    PUT    ?recurso=usuarios&id=N&accion=toggle-estado
    DELETE ?recurso=usuarios&id=N              (eliminar)

5.8 Modulo 8: Perfil de Usuario

  Funcionalidades:
    - Ver datos del usuario (nombre, correo, rol).
    - Cambiar contrasena (actual + nueva + confirmacion).
    - Validacion cross-field para coincidencia de passwords.

REALIZAR CON DIAGRAMAS
6. CASOS DE USO
================================================================================

CU01: AUTENTICACION Y ACCESO (LOGIN)
  Actor principal: Cualquier usuario.
  Precondiciones: El usuario debe existir en la BD con estado activo.
  Flujo principal:
    1. El usuario ingresa correo y contrasena.
    2. El sistema verifica las credenciales.
    3. Se genera un token JWT.
    4. El usuario es redirigido al dashboard segun su rol.
  Flujo de excepcion:
    - Credenciales incorrectas: se muestra "Credenciales invalidas".
    - Usuario inactivo: se muestra "Credenciales invalidas".
    - Campos vacios: se muestra "Email y password son obligatorios".

CU02: GESTION DE PROYECTOS
  Actor principal: Coordinador.
  Precondiciones: Estar autenticado como Coordinador.
  Flujo principal:
    1. Coordinador accede a "Nuevo Proyecto".
    2. Llena el formulario (nombre, descripcion, facultad, carrera, tutor, cupos).
    3. El sistema valida y guarda el proyecto.
    4. Se redirige a la lista de proyectos.
  Flujo de excepcion:
    - Campos invalidos: se muestran errores de validacion.
    - Tutor no valido: se muestra error.

CU03: INSCRIPCION EN PROYECTO
  Actor principal: Estudiante.
  Precondiciones: Estar autenticado como Estudiante, no estar en otro proyecto.
  Flujo principal:
    1. Estudiante ve la lista de proyectos.
    2. Hace clic en "Inscribirse" en un proyecto con cupos.
    3. Se crea una solicitud con estado "pendiente".
    4. Se muestra mensaje de exito.
  Flujo de excepcion:
    - Sin cupos: "No hay cupos disponibles".
    - Ya inscrito: "Ya estas inscrito en un proyecto".
    - Duplicada: "Ya existe una solicitud abierta".

CU04: REGISTRO Y APROBACION DE HORAS
  Actor principal: Estudiante (registro) / Tutor (aprobacion).
  Precondiciones: Estudiante inscrito en un proyecto con actividades.
  Flujo principal (registro):
    1. Estudiante selecciona una actividad.
    2. Ingresa las horas trabajadas (1-8).
    3. Se crea el registro con estado "pendiente".
  Flujo principal (aprobacion):
    1. Tutor accede a "Aprobar Horas".
    2. Ve los registros pendientes.
    3. Acepta o rechaza cada registro.

CU05: GESTION DE ACTIVIDADES
  Actor principal: Tutor.
  Precondiciones: Estar autenticado como Tutor con proyectos asignados.
  Flujo principal:
    1. Tutor accede a "Crear Actividad".
    2. Selecciona un proyecto (solo los suyos).
    3. Ingresa titulo, fecha y descripcion.
    4. Se crea la actividad.
  Flujo de excepcion:
    - Sin proyectos asignados: no aparecen opciones en el dropdown.

CU06: CRUD DE USUARIOS
  Actor principal: Coordinador.
  Precondiciones: Estar autenticado como Coordinador.
  Flujo principal:
    1. Coordinador accede a "Usuarios".
    2. Puede crear, editar, activar/desactivar o eliminar usuarios.
    3. Los cambios se reflejan inmediatamente en la tabla.

CU07: CAMBIO DE CONTRASENA
  Actor principal: Cualquier usuario.
  Precondiciones: Estar autenticado.
  Flujo principal:
    1. Usuario accede a "Mi Perfil".
    2. Ingresa contrasena actual, nueva y confirmacion.
    3. Si todo es valido, se actualiza la contrasena.

CU08: VISUALIZACION DE DASHBOARD
  Actor principal: Cualquier usuario.
  Precondiciones: Estar autenticado.
  Flujo principal:
    1. Al iniciar sesion, se muestra el dashboard.
    2. Las estadisticas varian segun el rol.
    3. Las tarjetas son clickeables para navegar a cada modulo.


7. FLUJOS GENERALES DEL SISTEMA - diagramas de actividades
================================================================================

7.1 Flujo de Autenticacion

  Usuario -> Login (correo + password)
    -> POST ?recurso=auth&accion=login
    -> Backend verifica credenciales
    -> Genera JWT (2h TTL)
    -> Frontend guarda token + usuario
    -> Redirige segun rol

7.2 Flujo de Inscripcion

  Estudiante -> Lista Proyectos -> "Inscribirse"
    -> POST ?recurso=solicitudes {id_proyecto}
    -> Backend valida: cupos, duplicada, proyecto activo, un proyecto por estudiante
    -> INSERT solicitud (estado: pendiente)
    -> Tutor/Coordinador -> Acepta/Deniega
    -> UPDATE solicitud + UPDATE cupos_usados (transaccion)

7.3 Flujo de Horas

  Estudiante -> "Registrar Horas" -> Selecciona actividad + horas
    -> POST ?recurso=horas
    -> Backend valida: actividad existe, horas 1-8
    -> INSERT horas_vinculacion (estado: pendiente)
    -> Tutor -> "Aprobar Horas" -> Acepta/Rechaza
    -> UPDATE horas_vinculacion (estado: aprobada/rechazada)


8. DISENO DE SEGURIDAD
================================================================================

  - Autenticacion: JWT con HMAC-SHA256, expiracion 2 horas.
  - Autorizacion: Dos capas (frontend RoleGuard + backend autorizarRolOrFail).
  - Proteccion SQL: Todas las consultas usan PDO prepared statements.
  - Proteccion de passwords: bcrypt via password_hash() / password_verify().
  - CORS: Configurado solo para http://localhost:4200.
  - Errores PHP: Suprimidos en produccion (error_reporting(E_ERROR|E_PARSE)).
  - Sesion: Token en localStorage, datos de usuario en sessionStorage.


9. CONCLUSION
================================================================================

El disenio detallado de la plataforma contempla 8 modulos funcionales, 8 casos
de uso principales, 8 tablas en base de datos (3FN) y una arquitectura MVC
separada en 3 capas. El sistema esta disenado para ser escalable, seguro y
facil de mantener, utilizando tecnologias modernas y estandares de la industria.

El disenio es inicial y esta sujeto a modificaciones segun las necesidades
detectadas durante la implementacion y las retroalimentaciones del equipo.
